Android HWUI 硬件加速原理解析：为什么不直接用 OpenGL 绘图？

在分析了 Bitmap 的内存分配（calloc 与物理内存映射产生的“首次触碰”惩罚）后，接下来的核心问题是：这些像素数据是如何被“画”到屏幕上的？

既然现代手机都有强大的 GPU，为什么 Android 不像游戏引擎（Unity/Unreal）那样直接使用 OpenGL/Vulkan 绘制纹理，而是构建了一套极其复杂的 HWUI 系统？

1. 渲染模式的本质区别

1.1 游戏引擎：即时模式 (Immediate Mode)

游戏引擎的核心通常是一个 while(true) 循环。每一帧，CPU 都会重新计算所有物体的位置，并向 GPU 发送全量的绘图指令（如 glDrawArrays）。

  • 特点：全量重绘。即使画面中只有一个按钮在闪烁，整个屏幕的像素通常也会重新计算。
  • 代价：极高的功耗，对手机电池极不友好。

1.2 Android UI：保留模式 (Retained Mode)

Android 采用的是“录制与回放”机制，将 UI 视作一棵**视图树 (View Tree)**：

  1. **录制 (Record)**：View 的 onDraw 并不直接产生像素，而是调用 DisplayListCanvas 将绘制动作（如 drawRect, drawBitmap）录制成 DisplayList（指令集）。
  2. **回放 (Playback)**：独立的 RenderThread（渲染线程） 持有这些指令，并决定如何调用 GPU 进行渲染。
  • 优点：只有当 View 发生变化（invalidate）时才重新录制。如果只是位移动画，RenderThread 可以直接复用旧指令，无需触发 Java 层重绘。

2. UI 系统的“阿喀琉斯之踵”：文本渲染

文本渲染是 UI 系统与游戏引擎最大的性能鸿沟，也是 Android 必须自研渲染引擎的主因。

  • 挑战：OpenGL 擅长画三角形和贴图，但极其不擅长处理复杂的矢量字形、抗锯齿、亚像素对齐和多国语言排版。
  • **HWUI 的解决方案 (Font Renderer)**：
    1. Skia 引擎：在 CPU 上精确计算字形的矢量路径。
    2. Texture Cache（灰度图纹理缓存）：HWUI 将渲染好的常用字符存储在一个巨大的 GPU 纹理缓存中。
    3. 合并绘制：当界面显示一段文字时，GPU 只是在屏幕上高速拼凑这些缓存好的小纹理块。
      这种“CPU 负责精细度 + GPU 负责合成”的混合模式，保证了文字在各种缩放级别下的极致清晰度。

3. 跨进程合成：SurfaceFlinger 与 HWC

Android 是一个多任务系统，屏幕上可能同时存在 App 窗口、状态栏、导航栏。

  • 独立性：每个窗口都是一个独立的 Surface。如果 App 直接写 OpenGL 到屏幕，系统将无法管理窗口重叠和透明度。
  • **Hardware Composer (HWC)**：
    • App 进程将自己的 UI 渲染到一个 GraphicBuffer 中。
    • SurfaceFlinger 进程收集所有 App 的 Buffer。
    • HWC 硬件在显示器输出前的一瞬间，直接在硬件层进行“图层叠加”。
      这种“零拷贝”机制避免了巨大的显存带宽浪费。

4. 为什么不直接用 OpenGL 纹理？

如果你尝试用原生 OpenGL 直接把整个 UI 绘制成一个大纹理，你会面临以下技术瓶颈：

4.1 功耗与脏区域 (Dirty Regions)

  • HWUI：支持局部更新。如果屏幕上只有一个进度条在动，HWUI 会计算出“脏区域”，只请求 GPU 重新渲染这 10% 的像素。
  • 纯 GL 模式：通常难以自动识别这种视图层级的局部变化，往往导致不必要的全屏刷新，显著缩短续航。

4.2 硬件抽象层：Skia 的价值

Android 通过 Skia 屏蔽了底层驱动的碎片化：

  • **Android 10+**：默认引入了 Skia Vulkan。
    开发者调用的 canvas.drawCircle()，到底层会自动转换为 GL 指令或 Vulkan 指令。这种架构让 Android 具备了在不同 GPU 硬件上保持一致表现的能力，而不需要开发者去处理复杂的 GPU 状态机。

5. 总结：HWUI 与 Bitmap 的关联

回到我们之前讨论的 Bitmap：

  1. 创建期：Bitmap 在 Native 层通过 calloc 分配，受限于 CPU 和内核映射速度。
  2. 上传期：当第一次绘制该 Bitmap 时，HWUI 会将其上传为 GPU Texture。
  3. 渲染期：一旦变成纹理，后续的平移、缩放、透明度变换都只是 DisplayList 中的几条简单指令，由 GPU 在几毫秒内完成，不再触发任何内存分配。

结论：Android 的 HWUI 并不是不用 OpenGL，而是为了 UI 场景的低功耗、高清晰度文本以及多任务安全性，在 GPU 之上封装了一套极其精密的“指令收集与分发系统”。

6. 延伸思考

既然 HWUI 这么强，为什么有些高级动画还是卡？

  • 原因：通常不是 GPU “画”得慢，而是因为动画过程中触发了 requestLayout() 导致全量 View 树遍历，或者频繁创建了大尺寸 Bitmap 触发了我们在前文中提到的“内核清零与缺页中断”惩罚。
  • 优化：应优先使用 RenderNode 或 Hardware Layer，让变化留在 GPU 内部。
